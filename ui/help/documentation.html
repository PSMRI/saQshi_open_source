SaQshi User Documentation

This guide explains what each page is used for and what actions a facility user can perform.

User Guide Developer Guide Assessment CQI Reports Performance Monitoring Certification AI Chat Assistant

Overview

SaQshi supports facility quality assessment, checklist scoring, continuous quality improvement, reporting, certification tracking, and monthly KPI/Outcome monitoring.

1. CreateCreate or continue an active assessment.
2. ActivateEnable applicable departments.
3. AssessFill assessor info and checklist responses.
4. ImproveUse gap analysis, action plan and closure.

Assessment Pages

Assessment List

Shows current and previous assessments for the facility with status and score.

  • Use Continue to open the checklist for in-progress assessment.
  • Review completed, cancelled and active assessment records.

Create Assessment

Create a new assessment only when no active assessment exists.

  • Enter assessment name, framework and date period.
  • If an active assessment exists, complete or cancel it before creating a new one.

Department Activation

Use this page to activate only departments applicable to the facility assessment.

  • Activated departments become available in Assessor Info, Checklist, KPI and Outcome workflows.
  • Activated department buttons become locked for the current assessment.
  • Inactive departments are not included in checklist entry.

Assessor Info

Capture assessment team details before checklist entry.

  • Select an activated department.
  • Enter assessor name, assessee name, date of assessment and assessment type.
  • If information already exists, the page shows saved details for review or update.

Checklist Entry

Checklist entry is performed one checkpoint at a time for selected department, area of concern and subtype.

  • Select department, area of concern, subtype and assessment method if available.
  • Score each checkpoint using 0, 1 or 2.
  • 0 means non-compliance, 1 means partial compliance, 2 means full compliance.
  • Use Next and Back to move between checkpoints.
  • If all checkpoints are completed, use Edit/Update to revise responses.

CQI: Gap Analysis, Action Plan and Gap Closure

PagePurposeUser Action
Gap Analysis Shows non-compliant and partially compliant checkpoints. Review gaps by department and area of concern before planning improvement.
Action Plan Creates action plans for every checkpoint gap. Select department/area/subtype, review suggested action plan, enter facility action plan, responsible person and target date.
Gap Closure Records closure evidence and revised score after action plan completion. Upload optional evidence, update revised score, remarks and closure status.
Final score is calculated using revised score where available; otherwise original checklist score is used.

KPI and Outcome Performance Monitoring

Performance pages capture monthly indicators for activated departments.

Indicator Entry

  • Select KPI or Outcome.
  • Select activated department and calendar month.
  • Enter numerator, denominator, result and remarks.
  • If denominator is N/A, denominator is locked and result is taken directly from numerator.
  • Use Save & Next to complete all indicators for the selected month.

Performance Dashboard and Trend

  • Dashboard shows total months, entries, KPI indicator count and Outcome indicator count.
  • Trend page shows month-wise KPI and Outcome chart views.
  • Use chart options to switch between line, area and bar views.
  • Use KPI/Outcome Excel buttons to download formatted performance reports.

Certification

The Certification module tracks State and National certification records for the logged-in facility.

Dashboard and History

  • Certification Dashboard shows State status, National status, next expiry date and renewal status.
  • Certification History lists previous State and National certification records.
  • Renewal Status highlights active, renewal due and expired certification records.
  • Report Dashboard includes a Certification Report shortcut that opens the Certification Dashboard.

Add Certification

  • Facility name and NIN load automatically from the logged-in user's assigned facility.
  • Facility ID is not shown on screen and is handled from the session.
  • Certification Type can be STATE or NATIONAL.
  • Assessment Mode can be PHYSICAL or VIRTUAL.
  • Applied Date is optional and cannot be greater than Certification Date.
  • Certification Date cannot be a future date.
  • CONDITIONAL certificates are valid for 1 year; CERTIFIED certificates are valid for 3 years.
  • National certification can be saved only after State certification exists.

AI Chat Assistant

The AI Chat Assistant is a floating help component available on authenticated dashboard pages.

How users use it

  • Open the floating chat button from the bottom-right corner.
  • Only one assistant button should be visible on the screen.
  • Ask questions about assessment, checklist scoring, CQI, KPI/Outcome, reports or certification.
  • Use quick suggestions such as Certification, Gap and Reports for common help.
  • Clear chat history when the conversation is no longer needed.

Current behavior

  • The assistant uses authenticated API endpoints under api/chat/v1.
  • Conversation history is stored per logged-in user and facility.
  • Responses are SaQshi workflow help generated by ChatAssistantService.php.
  • The active UI component is ui/components/chat-assistant; the older SQ.chat widget is disabled by default to avoid duplicate floating buttons.
  • The service boundary is ready for future external AI integration.

Reports

Report pages are used for assessment and performance review.

  • Report Dashboard: Summary of assessments, score, gaps, KPI months and Outcome months.
  • Score Report: Downloads full checklist scorecard in standard format.
  • Progress Report: Downloads checkpoint progress with action plan, responsible person, remarks and revised score.
  • KPI Report: Opens performance trend for KPI reports and Excel download.
  • Outcome Report: Opens performance trend for Outcome reports and Excel download.
  • Certification Report: Opens the Certification Dashboard from Report Shortcuts.

Profile and Facility Pages

My Profile

Facility user can update own name, mobile, email and password.

  • Password must include uppercase, lowercase, digit, special character and minimum 8 characters.
  • Profile fields are saved securely.

Facility Profile

Shows facility details from master data and allows update where permitted.

  • State, district, block and facility type are shown as names.
  • Geo coordinates can be captured from current location.
  • NIN number is checked for duplicate before update.

State, Regional, District and Block Monitoring

Monitoring users see the same monitoring tools, but SaQshi automatically limits data to the user's assigned level. Users do not need to choose this scope manually.

RoleRole IDData visible to userMain actions
State user9All facilities available in the configured state/facility master.Monitor all districts, certification, assessment progress, CQI, performance, facility drill-down, users and state reports.
Regional / Division user5Only facilities mapped to the user's assigned division or region.Use the same monitoring pages as state user, but counts, lists, maps and downloads are division-level only.
District user4Only facilities mapped to the user's assigned district.Review district facility category, certification status, assessment progress, CQI, performance and reports.
Block user8Only facilities mapped to the user's assigned block.Monitor block-level facilities, facility submissions, gaps, certification and performance entry status.

What changes after login?

The sidebar label changes automatically. A state user sees State Monitoring, a regional user sees Regional Monitoring, a district user sees District Monitoring and a block user sees Block Monitoring. The page links are shared, but the API filters the data by role scope.

  • Dashboard: shows total facilities, assessment started, in-progress and completed counts for the user's scope.
  • Certification Status and Map: show only certification records and map points inside the assigned state, division, district or block scope.
  • Facility Categorization: shows facility type and geography breakup only within the user's scope.
  • Assessment, CQI and Performance Monitoring: show facility-level progress only for facilities under the user's assigned geography.
  • State Reports: downloads are also scoped, so district/block users cannot download full state data.

Errors and Messages

If any page shows an error or warning, use this list to understand the reason and the next action.

Message / ErrorMeaningWhat user should do
Validation failed Required information is missing or wrong, for example facility ID, assessment ID, department, month or mandatory field. Check the selected facility, active assessment, department and form fields, then try again.
Facility ID is required The page could not identify the logged-in facility. Refresh once. If it still appears, logout and login again so facility session loads properly.
No active assessment No active assessment is available for checklist, department, KPI or Outcome workflow. Create a new assessment or continue the existing active assessment from Assessment List.
ACTIVE assessment already exists A new assessment cannot be created while another assessment is active. Complete or cancel the current assessment before creating a new one.
Already completed All checkpoints, action plans, closures, KPI or Outcome indicators are already entered for the selected scope. Use Edit/Update if changes are needed, otherwise move to the next department, month or report.
No data found No record exists for the selected filter, department, month or assessment. Change the filter or complete the related entry first.
Save failed The entered data could not be saved because of server, database or validation issue. Check required fields and internet/server connection, then save again. If repeated, contact support with page name and time.
404 Not Found The page or API URL is not available at the requested path. Open the page from the sidebar/menu instead of browser refresh on a wrong route. If API is missing, inform administrator.
500 Internal Server Error Server-side code failed while loading or saving data. Try once again. If it continues, share the page name, action and time with the technical team.
Session expired / Unauthorized Login session is missing or user is not allowed for that action. Login again. Use only pages assigned to the facility user role.
Duplicate NIN number The entered NIN number is already mapped with another facility. Verify the NIN number before updating Facility Profile.
Applied date cannot be greater than certification date The optional certification applied date is later than the selected certification date. Select an applied date on or before the certification date, or leave Applied Date blank.
Weak password Password does not match strength rule. Use at least 8 characters with uppercase, lowercase, digit and special character.
Captcha failed Captcha answer is wrong or expired on login. Enter the latest captcha shown on screen and login again.
Upload failed / invalid file Evidence file type, size or upload process is not accepted. Upload a valid image, PDF, Word, Excel or camera image. Remove wrong file and upload again if needed.
Location permission denied Browser did not allow current location for facility geo coordinates. Allow location permission in browser and try capture location again.
NaN / blank progress Score or progress cannot be calculated because total checkpoints or completed score is missing. Complete checklist entries first, then refresh dashboard/report.
When reporting an issue, mention page name, selected assessment, department/month, exact message and the time when the error appeared.

SaQshi Developer Guide

Technical specification, architecture, API and UI reference for developers maintaining the facility-user application.

Tech Specs Architecture API UI

Technical Specification

Technology Used

  • Frontend: HTML, CSS and vanilla JavaScript page bundles.
  • Backend: PHP versioned REST-style APIs under api/*/v1.
  • Data: MySQL tables, JSON framework/master files and uploaded evidence files.
  • Configuration: database credentials are loaded from local .env; use .env.example as the template and never commit real secrets.
  • Reports: PHP Excel generation for scorecard, progress, KPI and Outcome downloads.
  • Security: session auth, CSRF support, captcha login, password hashing, browser-side login password encryption and server-side encryption helpers.
  • Errors: database/PHP details are logged server-side; users receive friendly JSON messages with Request ID.

Framework and Master Data

  • NQAS/SaQshi checklist framework is loaded from api/config/frameworks/saqshi-nqas.json.
  • Departments, assessment methods, facility types and facility masters are under api/config/masters.
  • KPI, Outcome, formula, target and validation metadata are under api/config/performance.

Architecture

The application is organized as a lightweight facility-user portal with shared UI components, page-level modules and PHP service-backed APIs.

UI Layer

  • Each feature page normally has four files: .html, .js, .css and .json.
  • Shared layout and navigation are loaded from ui/components and ui/assets/js/core.
  • Facility-user pages are kept under ui/pages/facilityusers.

API and Service Layer

  • Versioned endpoints are grouped under api/<module>/v1.
  • Shared helpers are under api/core and business services under api/service.
  • JSON masters and framework data are under api/config.

Monitoring Role Scope

State, regional, district and block monitoring use the same ui/pages/state pages and api/state/v1 endpoints. Scope is applied centrally, not page-by-page.

  • role_id = 9 is State level and can see all configured facilities.
  • role_id = 5 is Regional/Division level and is filtered by session division_id.
  • role_id = 4 is District level and is filtered by session dist_id.
  • role_id = 8 is Block level and is filtered by session block_id.
  • api/state/v1/_bootstrap.php calls StateDashboardService::applyMonitoringScope(), which injects division, district or block filters into each request.
  • Scope names are resolved from api/config/masters/facilities.json, using the user's stored geography ids from s_user.

Login Password Transport

The login page does not send the plain password field to the login API. It first loads a public RSA key from api/auth/v1/login_key.php, encrypts the password in the browser with RSA-OAEP, then sends password_enc to api/auth/v1/login.php. The server decrypts it through api/core/LoginCrypto.php before normal password verification.

  • The private key is generated under api/storage/keys and this folder is ignored by git.
  • On Windows PHP, set OPENSSL_CONF in .env if key generation requires a local openssl.cnf path.
  • Developers may still see username, captcha and encrypted payload in browser DevTools, but the raw password should not appear in the request body.
  • This protects accidental password exposure during local debugging, but production deployments must still use HTTPS.
  • Shared API debug logging redacts sensitive keys such as password, password_enc, captcha and reset fields.
UI PageHTML/CSS/JS bundle handles screen and events.
API HelperJS calls versioned API endpoint.
PHP ServiceValidation, business rule and DB/config access.
ResponseJSON response updates UI state and messages.

Future Event Architecture / Kafka Readiness

Kafka is not recommended for the current simple deployment. Instead, keep the application ready for Kafka by introducing an event abstraction layer.

  • Use a single event entry point such as Event::dispatch("assessment.completed", $assessmentData);.
  • The abstraction is implemented in api/core/Event.php and loaded globally from api/bootstrap.php.
  • Current local event logs are written as JSON lines under api/storage/events/events-YYYY-MM-DD.log.
  • Every API that loads api/bootstrap.php automatically emits api.request.started and api.request.finished.
  • For now, Event::dispatch() can log events or call local PHP listeners.
  • Later, the same method can publish to Kafka without changing assessment, CQI, performance or certification business code.
  • Recommended future events: assessment.created, assessment.completed, department.activated, checklist.response.saved, gap.action_plan.saved, gap.closed, performance.kpi.saved, performance.outcome.saved and certification.updated.

This gives SaQshi an event-driven design path while keeping today's installation easy to run and maintain.
